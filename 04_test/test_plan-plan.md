# テスト計画書作成プラン

## 概要

E2UConverterのブラックボックステスト計画書を `04_test/test_plan.md` として作成する。
あわせてテストデータ生成スクリプト（Python）を `04_test/` に作成する。

テスト対象は実行可能JAR（`03_implementation/e2uconverter/target/e2uconverter-1.0.0.jar`）に対するコマンドライン実行ベースのブラックボックステストとする。

**コードページ**: IBM-930 をベーステストとし、IBM-1399 を追加テストとして設ける。

#### テストケース詳細の記述方針

各テストケースは以下の列を持つMarkdownテーブルで記述する:

| 列名 | 内容 |
|------|------|
| テストID | TG-X-NNN 形式 |
| テスト名 | 何をテストするか |
| 前提条件 | テストデータファイル名、オプション等 |
| 実行コマンド（例） | java -jar コマンド例（代表的なものは具体的に記述） |
| 確認項目 | 出力ファイル内容、レポート内容、標準出力、終了コードの期待値 |
| 合否基準 | 判定条件 |

#### 主要なテストケース観点

**TG-1: CLI引数検証（異常系）**
- FE-01: -i 未指定 → stderr に `ERROR: -i option is required.` , exit 1
- FE-02: -o 未指定 → stderr に `ERROR: -o option is required.` , exit 1
- FE-04: -l に文字列 → stderr に `-l option must be a positive integer.`
- FE-05: -m に不正値 → stderr に `-m option must be 'fixed' or 'stream'.`
- FE-06: 不正コードページ → stderr に `ERROR: Unsupported codepage: INVALID`
- FE-07: 存在しない入力パス → stderr にエラーメッセージ
- -l に 0 や負の値を指定 → エラー

**TG-2: 固定長レコードモード正常系**
- SBCSのみのソース（1レコード80バイト、複数レコード）
- DBCSを含むソース（SO-DBCS2バイト×N-SI）
- SBCS+DBCS混在ソース
- ファイルサイズがLRECLの倍数の場合（最終レコードがちょうど80バイト）
- ファイルサイズがLRECLの倍数でない場合（最終レコードが80バイト未満）
- 空ファイル（0バイト）→ 空のUTF-8ファイル出力
- -l 80以外のLRECL（例: 72, 133）

**TG-3: 固定長レコードモード 制御文字検知**
- 固定長モードで検知される制御文字の境界値テスト:
  - 0x00（最小値）
  - 0x0D（CR: 固定長モードでは不正）
  - 0x10, 0x3F（範囲の端）
  - 0xFF（変換不能）
- 制御文字を含むレコードで変換継続し `?` 置換されること
- レポートのセクション5に検知詳細が記録されること

**TG-4: 固定長レコードモード SO/SI不正検知**
- 正常なSO/SIペア（SO-DBCS2バイト×N-SI）
- SOなしでSI登場
- SIなしでSOがレコード末尾まで未クローズ
- SO直後にSI（空DBCS）
- SOのネスト（SO中にSO）
- 複数のSO/SIペアが1レコードに含まれる正常ケース

**TG-5: 固定長レコードモード 変換不能文字**
- 0xFF を含むレコード → `?` に置換、レポートに記録

**TG-6: バイトストリームモード正常系**
- SBCSのみ（NL=0x15区切り）
- SBCSのみ（CR=0x0D区切り）
- SBCSのみ（NL25=0x25区切り）
- 3種の行区切りコードが混在
- DBCS含む（SO/SI正常）
- SBCS+DBCS混在
- 行区切りなしでEOFのファイル（最後の行）
- 空ファイル

**TG-7: バイトストリームモード 制御文字検知**
- バイトストリームモードで検知される制御文字の境界値テスト:
  - 0x00〜0x0C
  - 0x10〜0x14
  - 0x16〜0x24
  - 0x26〜0x3F
  - 0xFF
- 0x0D, 0x15, 0x25 は行区切りとして処理され、不正文字として検知されないこと

**TG-8: バイトストリームモード SO/SI不正検知**
- TG-4と同様のパターン（1行単位での検証）
- 行をまたいだSO/SIは不正（前の行でSOが未クローズ）

**TG-9: バイトストリームモード 変換不能文字**
- 0xFF を含む行 → `?` 置換、レポート記録

**TG-10: 出力ファイルパス・拡張子決定**
- 拡張子なしファイル + -e未指定 → .txt付与
- 拡張子ありファイル + -e未指定 → 元の拡張子保持
- 拡張子なしファイル + -e指定 → 指定拡張子
- 拡張子ありファイル + -e指定 → 指定拡張子に置換
- 単一ファイル指定時の出力パス構造
- ディレクトリ指定時のサブディレクトリ構造の再現

**TG-11: ディレクトリ変換**
- -i にディレクトリ指定（-r なし）→ 直下ファイルのみ変換
- -i にディレクトリ指定（-r あり）→ サブディレクトリ含め再帰変換
- -r なしのとき、サブディレクトリ内ファイルは変換されないこと

**TG-12: レポート出力検証**
- -R 未指定 → <outputDir>/report.md に出力
- -R に相対パス指定 → 指定パスに出力
- -R に絶対パス指定 → 指定パスに出力
- レポートセクション1〜6の内容が仕様通りであること
  - セクション1: 処理日時・入力パス・出力先・コードページ・入力モード・再帰処理の表示
  - セクション2: 統計カウントの正確性（OK/WARNING/ERRORの件数）
  - セクション3: 全処理ファイルの一覧（ステータス・不正文字件数）
  - セクション4: エラーファイルなし時は「（なし）」表示
  - セクション5: 不正文字の行番号・オフセット・HEX値・理由が正確
  - セクション6: 256エントリのSBCSコード変換対応表

**TG-13: 致命的エラー処理**
- 各FE-01〜FE-07のエラー → exit code=1、stderrにメッセージ

**TG-14: ファイルエラー処理**
- 読み取り権限なしファイル → そのファイルはERROR、他ファイルは継続処理
- PE-01〜PE-03のエラーケース

**TG-15: IBM-1399コードページテスト**
- IBM-1399 は IBM-930 と異なりEBCDIK（日本語EBCDIC拡張）のコード体系を使用する
- -c IBM-1399 を指定した固定長モード正常変換（SBCS+DBCS混在）
- -c IBM-1399 を指定したバイトストリームモード正常変換（0x15区切り）
- IBM-1399 固有の文字が正しくUnicode変換されること（IBM-930と変換結果が異なるコードの確認）
- レポートのセクション6（SBCSコード変換対応表）にIBM-1399のコード体系が反映されること
- 不正文字検知の動作はIBM-930と同一であること（検知範囲はモードに依存し、コードページ非依存）

**Relevant Context**:
- 要件定義: `01_requirements/requirements.md`
- クラス設計: `02_design/02_class_design.md`
- エラー処理設計: `02_design/05_error_handling_design.md`
- 実行可能JAR: `03_implementation/e2uconverter/target/e2uconverter-1.0.0.jar`

**テストデータ定義の方針**:
- IBM-930 の主要なSBCS値（0x40=スペース, 0xF0〜0xF9=数字, 0xC1〜=英字等）を使用
- DBCSは 0x0E + 2バイト×N + 0x0F の構造
- 不正文字は境界値（0x00, 0x0D, 0x10, 0x3F, 0xFF等）を含める
- IBM-1399用データは、IBM-930と変換結果が異なるコードを意図的に含める
